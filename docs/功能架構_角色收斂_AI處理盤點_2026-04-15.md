# Campus Cloud 功能架構與角色收斂分析（2026-04-15）

## 1) 本次分析範圍

本文件聚焦三個問題：

1. 目前整體功能架構是否清楚且可運作。
2. 老師與同學功能配置是否已足夠收斂。
3. 是否存在「已具備能力但尚未真正落地」的功能。

限制：僅做架構與實作盤點，不修改程式邏輯。

---

## 2) 目前架構總覽（控制面 + AI 面）

### 2.1 控制面（主平台）

- 前端：React + TanStack Router/Query（`frontend/`）
- 後端：FastAPI + SQLModel + PostgreSQL（`backend/`）
- 資源控制：Proxmox API（VM/LXC、快照、監控、網路）
- 角色主軸：student / teacher / admin（另有 is_superuser）

### 2.2 AI 面（實際上是三條並行鏈）

- 鏈 A：申請輔助與模板推薦
  - 入口：`/api/v1/ai/template-recommendation/*`
  - 前端使用：`ApplicationRequestPage` 內的 `AiChatPanel`
  - 上游：vLLM（由 backend 設定指向）

- 鏈 B：PVE 佈署建議
  - 入口：`/api/v1/ai/pve-advisor/recommend`
  - 主要由 backend VM 排程/可用性流程內部使用
  - 前端沒有直接「可視化結果頁」

- 鏈 C：對外 AI API Proxy（OpenAI 相容）
  - 入口：`/api/v1/ai-proxy/*`
  - 權杖流：`/api/v1/ai-api/*`（申請/審核/憑證）
  - 速率限制：Redis sliding window（可關閉）

### 2.3 AI 處理「機器」現況判讀

目前不是單一 AI 引擎，而是「多入口 + 可外接上游模型」架構：

- 主平台 backend 內有 AI 路由與服務。
- 推論機可以是 `vllm-inference/` 或 `vllm-API/`，但未被 root compose 直接納入。
- 換句話說，AI 推論平面與主平台控制平面是鬆耦合部署。

這種設計彈性高，但需要更強的配置治理（見第 6 節建議）。

## 3) 老師 vs 同學：收斂度評估

## 結論（先講重點）

「部分收斂，但尚未閉環」。

- 同學功能流（申請 → 使用資源 → AI API 申請）相對完整。
- 老師功能流（群組管理/批量佈署）有做，但在「教學審閱閉環」上有明顯斷點。

可用一行概括：

- 同學：可完成主要任務。
- 老師：可管理群組，但關鍵教學輔助流程有角色閘門不一致。

### 3.1 已收斂的部分

- 老師比同學多了群組管理與批量建立資源能力。
- 老師/管理員有 immediate mode（可自動審核與立即部署）。
- 同學、老師、管理員都可走 application 流程（差在後續權限）。

### 3.2 未收斂（高優先）

1. AI 評分助手路徑前後端權限不一致
- 前端路由 `groups/$groupId/ai-judge` 僅允許 admin。
- 但 backend `rubric` API 是 InstructorUser（Teacher/Admin 可用）。
- 結果：老師看得到入口語意，但實際被前端 gate 擋掉，教學流程中斷。

2. 群組頁導向的資源詳情路徑對老師不友善
- 群組成員 VM 狀態可見，且連到 `/resources/$vmid`。
- 老師通常不是資源 owner，後端資源細節依 owner/admin 檢查。
- 結果：老師可能在群組場景點進資源細節時被拒絕（體感像功能壞掉）。

3. 老師角色定位仍偏「群組操作員」，不是「教學審核者」
- VM requests 審核頁面與 review API 都是 admin 主導。
- 老師在申請治理上的參與點，主要靠 immediate mode，而非完整審核工作流。

## 4) 可以利用但尚未真正做完的能力

以下是「已有程式能力或端點，但落地度不足」：

1. Script Deploy 能力疑似半成品
- 後端有完整 `/script-deploy` 路由（deploy/status/register）。
- 前端有 `ScriptDeployDialog` 元件，但實際路由整合不足，存在註解與現況不一致跡象。
- 判定：能力存在，但產品入口與流程未完成收斂。

2. PVE Advisor 雖可呼叫，但缺少前端可解釋視圖
- 前端可看到 OpenAPI 生成型別，但缺少面向老師/管理員的「建議可視化頁」。
- 目前多偏後端內部決策使用。
- 判定：AI 能力有，但可觀測性與信任建立不足。

3. AI 設定治理分散
- `template_recommendation` / `pve_advisor` 用 `backend/config/system-ai.json`。
- `rubric` 仍依賴另一套 env 設定命名。
- `ai-proxy` 又有自己 `features/ai/config`。
- 判定：可運作，但配置面碎片化，對「AI 機器」切換與維運不夠友善。

4. root compose 未納入 AI 推論服務
- 主 stack 起來後，AI 能力仍需額外啟動/配置上游。
- 判定：開發者可處理，但對教學現場的一鍵可用性仍不足。

## 5) 角色收斂評分（給決策者看的版本）

- 同學流程完整度：8/10
- 老師流程完整度：6/10
- 老師/同學收斂一致性：6.5/10
- AI 能力可用性：7/10
- AI 能力可營運性（配置/部署/觀測）：5.5/10

整體判讀：

- 「功能多」但「角色體驗閉環」還沒完成。
- 下一步應該優先做角色與入口一致化，而不是再加新功能。

---

## 6) 優先建議（不改架構大方向）

### P0（先修斷點，1~2 週）

1. 對齊 AI Judge 權限策略
- 若後端設計是 Teacher 可用，前端 route guard 應同步。
- 若產品要 admin-only，則 backend 也應同步收緊並在群組頁隱藏入口。

2. 修正群組頁的資源導向體驗
- 老師點群組成員 VM 時，導向「教師可讀」頁（或只顯示可操作摘要），避免落入 owner-only 細節 API。

### P1（角色產品化，2~4 週）

1. 定義 Teacher persona 的完整閉環
- 建議明確決策：老師是否參與申請審核。
- 若參與，補 teacher-friendly review queue；若不參與，UI 上不要暗示可審核。

2. 把 PVE Advisor 結果做成可解釋視圖
- 顯示推薦節點、資源壓力、風險指標、AI 介入比重。
- 讓老師/管理員看得懂「為什麼放這台」。

### P2（AI 維運收斂，4~6 週）

1. 統一 AI 設定來源
- 建議收斂成單一設定模型（例如 system-ai + env overlay），避免三套路徑並存。

2. 明確 AI 推論平面標準
- 建議選定單一「正式上游」（如 `vllm-API`）作為統一入口。
- 其他服務視為實驗/benchmark，避免環境漂移。

3. 補上運維可視化
- 模型可用性、延遲、token throughput、錯誤率、配額使用應可在管理頁查看。

---

## 7) 決策摘要

- 問題不是「功能太少」，而是「角色體驗與權限邏輯尚未完全收斂」。
- AI 能力其實不少，但入口一致性、配置治理、可視化說明仍不足。
- 若目標是教學現場穩定落地，建議先做 P0/P1 收斂，再擴 AI 新功能。
